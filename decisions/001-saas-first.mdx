## Context

The spec as originally drafted was local-first: a developer scaffolded a template, files lived in a git repo, and `bun dev` ran the browser app. Cloud services (Fastify + Postgres) were mentioned only for Phase 4 player save sync.

However, Alumic's product framing — "browser-first creative tools platform", Articy:Draft killer, canvas + UI designer + vector editor all in the browser — implies a hosted experience. A designer should be able to visit alumic.app, create a project, and have their work persist to an account. None of that was architected.

The two realistic positions were (a) local-first only, defer SaaS, or (b) SaaS-first with on-prem as an enterprise option. Larger studios will insist on on-prem for IP security and compliance, but hosted SaaS is the primary revenue and adoption path.

## Decision

Alumic ships as a SaaS product first. The hosted deployment (alumic.app) and the on-prem deployment run from a single codebase; deployment mode is selected by configuration. Enterprise-only features (SSO, audit logs, team management depth) are gated by license/config flags, not by separate forks.

## Consequences

**Enables:** Browser-only authoring workflow. Team collaboration, live rebalancing via entity edits, managed backups, and a marketplace ([ADR-021 Templates and plugins distributed via Alumic marketplace (Phase 4)](/decisions/021-marketplace-phase-4)) all become natural. On-prem customers get the full stack as a Docker image or bun binary.

**Obligates:** Backend infrastructure is no longer a Phase 4 concern — it is the primary data layer. Interfaces (EntityDatabase, plugin loading, save sync) must be designed async/remote-friendly from day one. Authored files (`.graph.json`, `.entity.json`) become export/import/on-prem seeding format, not primary storage.

**Forecloses:** A pure local-first product with optional cloud sync. Users who want "my files on my disk" workflows use the on-prem deployment or file export.
