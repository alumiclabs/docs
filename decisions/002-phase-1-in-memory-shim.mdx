## Context

[ADR-001 SaaS-first, on-prem second](/decisions/001-saas-first) commits Alumic to SaaS-first, which normally implies shipping the backend (Fastify + Postgres + entity CRUD API + auth) before anything else. But Phase 1's milestone — "`bun dev` produces a running browser scene with a moving cube" in 6-8 weeks — is achievable only if the kernel, renderer, input, and first template can be built and tested without backend infrastructure in place.

The options: (a) extend Phase 1 by 4-6+ weeks to include the backend skeleton, (b) defer SaaS-shaped interfaces and rewrite in Phase 2, or (c) design async-remote-friendly interfaces from day one, ship Phase 1 with an in-memory implementation behind those interfaces, then drop in the real backend in Phase 2.

## Decision

Phase 1 ships with an in-memory (optionally IndexedDB-persisted) implementation of `EntityDatabase` behind the same interface the Phase 2 backend will implement. Reads and writes are async throughout. Phase 2 replaces the implementation with Fastify + Postgres; zero rewrite cost at the call site.

## Consequences

**Enables:** Phase 1 ships a runnable framework on schedule without premature backend scope. Interface discipline (async, no synchronous file I/O, no module-local state that presumes single-process) is forced from the start.

**Obligates:** Phase 1 interface design must be reviewed against the expected Phase 2 backend shape (pagination, optimistic concurrency tokens, streaming query support if needed). Otherwise the shim ships an interface that will leak when it meets the real backend.

**Neutral:** Phase 1 remains offline-capable; the shim works without a network. Tests use the shim by default.
