## Context

[ADR-034 Marketplace bundles are signed; third-party plugins execute in a worker sandbox](/decisions/034-marketplace-sandbox-signing) introduced `networkAllowlist` on third-party plugin meta to gate outbound requests. The enforcement mechanism was not specified. Web Workers have unrestricted `fetch` by default; a declared allowlist with no enforcement primitive is theater. Two options existed: wrap all plugin fetches in a host-realm RPC proxy, or serve plugin bundles from per-plugin origins with a Content-Security-Policy header the browser enforces.

## Decision

Third-party plugin bundles are served from `https://plugins.alumic.app/<pluginId>@<version>/` (and the equivalent on-prem hostname). The response for `bundle.js`, `worker.js`, and any sandboxed iframe documents carries a CSP header:

```
Content-Security-Policy:
  default-src 'none';
  script-src 'self';
  connect-src 'self' https://api.alumic.app <networkAllowlist entries>;
  img-src 'self' https://assets.alumic.app blob: data:;
  worker-src 'self';
```

`<networkAllowlist entries>` is composed from the plugin's declared `networkAllowlist`. The CSP is generated by the publish pipeline and attached to the signed bundle manifest; the edge server emits it from the manifest so changes ride with the bundle, not with a deploy.

`ctx` access from the Worker is still marshaled through `postMessage` RPC to the host realm ([ADR-034 Marketplace bundles are signed; third-party plugins execute in a worker sandbox](/decisions/034-marketplace-sandbox-signing)). CSP enforces the **outbound network boundary**; the RPC proxy enforces the **Alumic-API scope boundary**. The two layers are independent: a plugin cannot call `/api/projects/other/entities` even if its CSP allows `api.alumic.app`, because the host-realm RPC enforces projectId scoping.

First-party plugins remain in the main realm with the editor's own CSP; they are trusted at the same level as the Alumic code they run alongside.

## Consequences

**Enables:** Declared `networkAllowlist` is browser-enforced at no runtime cost. On-prem inherits the model by configuring a plugin-serving hostname. The host realm never needs a fetch-proxy primitive.

**Obligates:** The edge layer (CDN or reverse proxy) emits per-bundle CSP headers — either by serving the bundle through an Alumic-controlled worker that reads the signed manifest, or by baking CSP into the storage object metadata. Per-plugin subdomains require wildcard TLS on `plugins.alumic.app`.

**Forecloses:** Loading third-party plugin bundles from arbitrary origins (consistent with [ADR-034 Marketplace bundles are signed; third-party plugins execute in a worker sandbox](/decisions/034-marketplace-sandbox-signing)). A plugin with no `networkAllowlist` declaration gets `connect-src 'self' https://api.alumic.app` — it can still talk to the Alumic API, nothing else.
