> **Three disjoint "projects" вЂ” don't conflate them.** This chapter documents
> the **Alumic monorepo** layout used by framework developers. It is not the
> layout of a template-author's package, and it is not the layout of an
> end-user game project:
>
> - **(a) Alumic monorepo** (this chapter) вЂ” the repo Alumic itself is built
>   in. Contains `@alumic/*` packages, first-party templates, and CLI tooling.
> - **(b) Template package** ([Template System](/reference/template-system)) вЂ” a Bun workspace package authored by a
>   template author and published to the marketplace ([ADR-021 Templates and plugins distributed via Alumic marketplace (Phase 4)](/decisions/021-marketplace-phase-4)). Seed entities
>   ship inside it; the publish pipeline ([ADR-033 Engine-portable transpile runs server-side on publish](/decisions/033-engine-transpile-server-side)) ingests them into the
>   backend at install time.
> - **(c) End-user game project** ([ADR-025 Project identity is a server-issued UUID](/decisions/025-project-identity-server-uuid), [ADR-035 CLI is SaaS-aware by default](/decisions/035-cli-saas-aware)) вЂ” a server-side row with
>   a server-issued UUID, owned by a workspace. Under SaaS-first it has
>   **no local `data/` tree** by default; `alumic pull` / `alumic export`
>   writes an optional working copy for Git workflows and offline authoring.
>
> Sections below describe (a). See [Template System](/reference/template-system) for (b) and [ADR-025 Project identity is a server-issued UUID](/decisions/025-project-identity-server-uuid)/[ADR-035 CLI is SaaS-aware by default](/decisions/035-cli-saas-aware) for (c).

## Monorepo Layout

Alumic uses a **Bun workspace monorepo**. All packages, templates, and tooling live in a single repository with shared build configuration.

**Why Bun over pnpm + Turborepo** (which the research recommended): Alumic's desktop distribution runs on [Electrobun](https://electrobun.dev), a lightweight native shell (~6 MB) driven by a Bun runtime. Keeping Bun as the runtime from dev through desktop packaging means a single toolchain instead of layering Bun on top of a pnpm/Turborepo setup. Bun workspaces provide native TypeScript execution, a built-in test runner, and `workspace:*` semantics equivalent to pnpm's. If incremental-build caching becomes painful, external caching (Nx Cloud or content-addressable build outputs) can be added without reintroducing Turborepo. See [ADR-005 Bun workspaces as the monorepo tool](/decisions/005-bun-workspaces).

## Per-Package Conventions

### `package.json` (example: @alumic/core)

```json
{
  "name": "@alumic/core",
  "version": "0.0.1",
  "type": "module",
  "main": "./src/index.ts",
  "types": "./src/index.ts",
  "exports": {
    ".": {
      "import": "./src/index.ts",
      "types": "./src/index.ts"
    }
  },
  "scripts": {
    "build": "tsc -b",
    "clean": "rm -rf dist",
    "test": "bun test"
  },
  "dependencies": {
    "@alumic/tags": "workspace:*"
  },
  "peerDependencies": {},
  "devDependencies": {
    "typescript": "^5.7"
  }
}
```

**Key conventions:**
- `"type": "module"` вЂ” ESM only, no CommonJS
- `"main"` and `"exports"` point to **source** (`./src/index.ts`) during development, enabling direct TS imports via Bun's native TypeScript support
- For publishing to npm (future), a build step will produce `./dist` with compiled JS + `.d.ts` declarations, and `exports` will be updated to point there
- `workspace:*` protocol for inter-package dependencies
- No bundler needed during development вЂ” Bun resolves TypeScript imports natively

### Directory Layout Within a Package

```
packages/example/
```

**Rules:**
- One concept per file (e.g., `plugin.ts` defines `AlumicPlugin` + `PluginRegistry`)
- `index.ts` is a barrel that re-exports only the public API вЂ” internal helpers stay private
- Tests live in `__tests__/` adjacent to `src/`, not co-located with source
- No `src/internal/` exports in `index.ts`

## Build Pipeline

### Development (Bun native)

During development, no build step is needed. Bun natively resolves TypeScript imports, including cross-workspace references via `workspace:*`. Running `bun run src/index.ts` in any package "just works."

For templates, `bun --hot` or Vite dev server provides hot reload.

### Type Checking

```bash
# Check all packages via project references
bun run typecheck    # в†’ tsc -b

# Check a single package
cd packages/core && tsc --noEmit
```

### Testing

```bash
# Run all tests from root
bun test

# Run tests for a specific package
bun test packages/core

# Watch mode
bun test --watch
```

Vitest is configured at the root for consistent test behavior across packages. Bun's built-in test runner (`bun test`) is also supported for simpler cases.

### Production Build (Future)

When packages need to be published to npm or bundled for production deployment:

1. `tsc -b` compiles TypeScript to JavaScript with declarations
2. Output goes to `./dist` per package
3. `package.json` exports are updated to point to `./dist`
4. Tree-shaking is handled by the consumer's bundler (Vite, esbuild, etc.)

For templates, Vite or Bun's built-in bundler produces optimized static assets.

## Dependency Management

### Internal Dependencies

Packages reference each other using Bun's workspace protocol:

```json
{
  "dependencies": {
    "@alumic/core": "workspace:*",
    "@alumic/tags": "workspace:*"
  }
}
```

The `workspace:*` specifier resolves to the local workspace package during development. When published, it resolves to the actual version number.

### External Dependencies

- **Three.js**: Declared as a `peerDependency` in `@alumic/render-three` with a version range (e.g., `"three": ">=0.170.0"`). The monorepo root pins a specific version.
- **SolidJS**: Declared as a dependency in template packages and any UI-related packages.
- **Zod**: Declared as a dependency in packages that define schemas (`@alumic/core`, `@alumic/dialogue`, etc.).

### Avoiding Dependency Bloat

- `@alumic/tags` has zero external dependencies
- `@alumic/core` depends only on `@alumic/tags` and `zod`
- `@alumic/render` is types-only вЂ” zero runtime dependencies
- Feature plugins declare the minimum necessary dependencies
- Three.js is never imported by `@alumic/core` вЂ” only by `@alumic/render-three`

## Project Identity

A user-created project is not a directory вЂ” it is a server-issued UUID ([ADR-025 Project identity is a server-issued UUID](/decisions/025-project-identity-server-uuid)). When a file-shaped workspace exists on disk (on-prem seed, on-prem export, CLI `--offline`), its root carries `project.json`:

```json
{
  "$format": "alumic-project",
  "$version": 1,
  "id": "0b7e5f1c-3e8a-4a3a-9a5e-1f9b7c4b8d21",
  "name": "Forest Game",
  "workspaceId": "01HR0K...",
  "createdAt": "2026-04-20T12:00:00.000Z"
}
```

CLI commands (`alumic push`, `alumic pull`, `alumic export`) resolve the project by reading this file. Directories without `project.json` are anonymous until pushed, at which point a UUID is minted and the file is stamped.

## Environment Variables

Template dev servers and the backend use `.env` files for deployment-specific configuration (backend URL for the browser, database URL for the backend). These are never committed.

```

VITE_ALUMIC_BACKEND=http://localhost:3000

# packages/backend/.env.local (gitignored)
ALUMIC_DEPLOYMENT=desktop          # saas | onprem | desktop
DATABASE_URL=postgres://localhost:5432/alumic
OBJECT_STORAGE_ENDPOINT=http://localhost:9000
JWT_SIGNING_KEY=...
```
